# API reference

The full interactive reference is auto-generated by FastAPI at [`/api/docs`](http://localhost/api/docs) once the stack is running.

This file is a stable, human-readable index that we keep updated as endpoints are added.

## Conventions

- All paths in this doc are written **without** the `/api` Caddy prefix. From a browser/client they're called as `/api/auth/login`; from inside the FastAPI app they're `/auth/login`.
- Auth is via the `hiremesh_session` cookie (see [auth.md](./auth.md)).
- Errors return `{"detail": "..."}` with an appropriate HTTP status.

## M2 endpoints

### Resumes

| Method | Path | Auth | Notes |
|---|---|---|---|
| `POST`   | `/candidates/{id}/resumes` | cookie | multipart `file` (PDF/DOCX, ≤10 MB). Enqueues parse. |
| `GET`    | `/candidates/{id}/resumes` | cookie | List, primary first |
| `POST`   | `/resumes/{id}/primary` | cookie | Make this resume the primary |
| `POST`   | `/resumes/{id}/reparse` | cookie | Resets to `pending`, re-enqueues |
| `DELETE` | `/resumes/{id}` | cookie | Deletes object + row; promotes next-newest to primary |
| `GET`    | `/resumes/{id}/url` | cookie | Returns `{url, expires_in}` — pre-signed download URL (5 min TTL) |

`Resume` shape: `{id, candidate_id, filename, mime, is_primary, parse_status, parse_error?, parsed_json?, created_at}`. `parse_status` ∈ `pending | parsing | done | failed`.

### Behavior changes elsewhere in M2

- `PATCH /candidates/{id}` now records every changed field in `candidate_field_overrides` so the parser cannot stomp manual edits. See [resumes-and-parsing.md](./resumes-and-parsing.md#the-sticky-edit-invariant--the-rule-that-keeps-human-edits-safe).

## M3 endpoints

### Pipeline (kanban + audit)

| Method | Path | Auth | Notes |
|---|---|---|---|
| `GET`    | `/jobs/{id}/board` | cookie | Stages with their candidate-jobs (and embedded candidate). Soft-deleted candidates excluded. |
| `POST`   | `/jobs/{id}/candidates` | cookie | `{candidate_id}` — link, auto-placed at stage 0. `409` if already linked. |
| `PATCH`  | `/candidate-jobs/{id}` | cookie | `{stage_id}` — move; writes a `stage_transitions` row. Stage must belong to the same job. |
| `DELETE` | `/candidate-jobs/{id}` | cookie | Unlink; writes a final `to_stage_id=NULL` audit row. The transitions persist. |
| `GET`    | `/candidate-jobs/{id}/transitions` | cookie | Full history for this link. |

| `POST`   | `/candidate-jobs/{id}/notes` | cookie | Add a link-scoped note. |

`Note` shape gains a nullable `candidate_job_id`. See [pipelines.md](./pipelines.md#notes-scope-reminder) for what global vs link-scoped means.

## M4 endpoints

### Search

| Method | Path | Auth | Body | Notes |
|---|---|---|---|---|
| `POST` | `/search/candidates` | cookie | `SearchRequest` | Returns `SearchHit[]`. Filters always apply; `q` adds semantic ranking |

`SearchRequest`:
```json
{
  "q": "string",                       // optional; if empty → filter-only
  "location": "string",
  "skills": ["python", "kafka"],
  "exp_min": 3, "exp_max": 8,
  "stage_name": "Interested",          // candidate is currently at this stage on any job
  "limit": 50, "offset": 0
}
```

`SearchHit`:
```json
{ "candidate": { /* CandidateOut */ }, "score": 0.823 }
```

`score` is cosine similarity (0–1) when `q` is present; `null` for filter-only.

### Admin

| Method | Path | Auth | Notes |
|---|---|---|---|
| `POST` | `/admin/reindex/candidates` | admin | Enqueues an embed task for every active candidate. Returns `{enqueued: <count>}` |

## M5 endpoints

### Ask

| Method | Path | Auth | Body | Returns |
|---|---|---|---|---|
| `POST` | `/ask/candidate/{id}` | cookie | `{question}` | `{answer, citations: [{type, id, snippet}]}` |
| `POST` | `/ask/pool` | cookie | `{question}` | `{answer, route, citations, rows?, matched_count}` |

Pool answer fields:
- `route` ∈ `structured | semantic | hybrid` — which path the classifier picked
- `rows` (structured only) — raw rows the SQL query returned
- `citations` (semantic/hybrid) — candidates the answer references, each with `id` so the UI can link to `/candidates/{id}`
- `matched_count` — total candidates matched (semantic/hybrid) or row count (structured)

`citations[].type` ∈ `profile | resume | note | row` (per-candidate uses the first three; pool uses `row`).

See [search-and-ask.md](./search-and-ask.md#m5--ask) for the routing logic and the safety boundary on SQL generation.
